# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문

## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.

## 프로젝트 전체

- 이 서비스가 해결하는 문제는 무엇인가?

  → CareerFit AI는 취업과 공모전을 준비하는 대학생이 자신의 전공, 보유 스킬, 관심 직무를 바탕으로 어떤 역량을 더 준비해야 하는지 판단하기 어려운 문제를 해결하기 위한 서비스이다. 채용 공고 데이터를 기반으로 사용자에게 맞춤형 역량 분석과 포트폴리오 준비 방향을 제공한다.

- RAG를 사용하지 않으면 어떤 문제가 생기는가?

  → RAG를 사용하지 않으면 AI가 실제 채용 공고 데이터를 참고하지 않고 일반적인 답변만 생성할 가능성이 높다. 이 경우 사용자의 입력과 실제 공고 요구사항이 잘 연결되지 않을 수 있고, 존재하지 않는 정보나 근거 없는 추천이 나오는 Hallucination 문제가 커질 수 있다.

- 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?

  → 가장 어려웠던 구현은 FastAPI 백엔드, ChromaDB 검색, Gemini API, React 프론트엔드를 하나의 흐름으로 연결하는 부분이었다. 처음에는 각각의 기능이 따로 동작했지만, `/analyze` API에서 사용자 입력을 받아 RAG 검색을 수행하고, 검색 결과를 Gemini 프롬프트에 넣어 답변과 sources를 함께 반환하도록 구성하면서 해결했다. 이후 React에서 fetch로 API를 호출하고 ResultCard와 SourceCard에 결과를 출력하도록 연결했다.

## 데이터 관련

- 왜 이 데이터를 선택했는가?

  → CareerFit AI는 취업·공모전 준비를 돕는 서비스이기 때문에, 사용자의 전공·스킬·관심 직무와 직접 연결될 수 있는 채용 공고 데이터를 선택했다. 채용 공고에는 기업이 실제로 요구하는 직무, 기술 스택, 마감일 정보가 포함되어 있어 역량 분석과 포트폴리오 방향 추천에 적합하다고 판단했다.

- 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?

  → 채용 공고 데이터는 기업마다 작성 방식이 다르기 때문에 요구 스킬이 누락되거나 과장되어 있을 수 있다. 또한 마감일, 직무명, 기술 스택 등이 일관된 형식으로 작성되지 않을 수 있고, 오래된 공고라면 현재 채용 시장 상황을 정확히 반영하지 못할 수 있다. 따라서 데이터 전처리와 최신성 관리가 필요하다.

- SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?

  → SQLite는 공고 데이터를 표 형태로 안정적으로 저장하고 조회하기 위한 구조화 데이터베이스이고, ChromaDB는 자연어 검색과 RAG를 위한 벡터 데이터베이스이다. 같은 원본 데이터를 사용하더라도 SQLite는 정리된 데이터 관리용, ChromaDB는 사용자 질문과 유사한 공고를 찾는 검색용으로 역할이 다르기 때문에 둘 다 사용했다.

- 결측치와 중복을 어떻게 처리했는가?

  → Pandas를 사용해 각 컬럼의 결측치 개수와 비율을 먼저 확인했다. 서비스에 꼭 필요한 필수 컬럼에 결측치가 있는 경우 해당 행을 제거했고, 선택적으로 비어 있어도 되는 값은 빈 문자열 등으로 처리했다. 중복 공고는 회사명, 공고명, 직무 유형 등 주요 정보를 기준으로 중복 여부를 확인한 뒤 제거했다.

## RAG 관련

- RAG 구조에서 ChromaDB는 어떤 역할을 하는가?

  → ChromaDB는 채용 공고 데이터를 벡터 형태로 저장하고, 사용자의 입력과 의미적으로 유사한 공고를 검색하는 역할을 한다. 사용자가 전공, 스킬, 관심 직무를 입력하면 ChromaDB가 관련성이 높은 공고를 찾아주고, 이 검색 결과가 Gemini 답변 생성의 근거로 사용된다.

- n_results 값을 높이면 어떤 트레이드오프가 생기는가?

  → n_results 값을 높이면 더 많은 공고를 참고할 수 있어 정보량이 늘어나는 장점이 있다. 하지만 관련성이 낮은 문서까지 함께 들어갈 수 있어 답변이 흐려지거나 불필요하게 길어질 수 있다. 또한 LLM에 전달되는 context가 길어져 처리 시간이 늘어나고 비용이나 토큰 사용량도 증가할 수 있다.

- 검색 결과 품질을 어떻게 검증했는가?

  → 여러 테스트 쿼리를 직접 입력해 검색 결과가 사용자의 관심 직무와 관련 있는지 확인했다. 예를 들어 “통계학과 Python SQL 데이터 분석” 같은 입력을 넣었을 때 데이터 분석 관련 공고가 상위에 검색되는지 확인했다. 또한 검색 결과의 distance 값을 참고해 입력과 가까운 문서가 잘 선택되는지 검토했다.

- Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?

  → Hallucination은 AI가 실제로 존재하지 않는 정보나 근거 없는 내용을 사실처럼 생성하는 현상이다. RAG는 AI가 답변을 만들기 전에 실제 데이터베이스에서 관련 문서를 검색하고, 그 문서를 근거로 답변하게 만든다. 따라서 모델이 기억이나 추측만으로 답하는 상황을 줄이고, 실제 공고 데이터에 기반한 답변을 생성하도록 도와준다.

## API 관련

- /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?

  → `/analyze`는 사용자의 전공, 보유 스킬, 관심 직무처럼 여러 입력 데이터를 요청 본문에 담아 서버로 보내야 하기 때문에 POST를 사용했다. GET은 주로 단순 조회에 사용되고, POST는 사용자의 입력을 기반으로 분석 결과를 생성하는 요청에 적합하다.

- 응답 JSON에 sources를 포함시킨 이유는 무엇인가?

  → sources를 포함하면 AI 답변이 어떤 공고 데이터를 근거로 생성되었는지 사용자가 확인할 수 있다. 이는 답변의 신뢰도를 높이고, 단순한 AI 조언이 아니라 실제 데이터 기반 분석이라는 점을 보여준다. 또한 디버깅이나 검색 품질 확인에도 도움이 된다.

- /health 엔드포인트가 왜 필요한가?

  → `/health` 엔드포인트는 서버가 정상적으로 실행 중인지 빠르게 확인하기 위해 필요하다. 배포 환경이나 Docker 실행 후에도 `/health`에 접속해 백엔드가 살아 있는지 확인할 수 있다. Render 같은 배포 환경에서도 서비스 상태 점검용으로 활용할 수 있다.

## 보안 관련

- .env 파일을 GitHub에 올리면 어떤 일이 생기는가?

  → `.env` 파일에는 Gemini API Key 같은 민감한 정보가 들어갈 수 있다. 이 파일을 GitHub에 올리면 API Key가 외부에 노출되어 다른 사람이 무단으로 사용할 수 있고, 비용 발생이나 계정 보안 문제가 생길 수 있다. 따라서 `.env`는 반드시 `.gitignore`에 포함해야 한다.

- .env와 .env.example의 차이는 무엇인가?

  → `.env`는 실제 API Key와 환경변수 값이 들어가는 개인 설정 파일이고, GitHub에 올리면 안 된다. 반면 `.env.example`은 어떤 환경변수가 필요한지 알려주는 예시 파일이다. 실제 키 대신 `your_api_key_here` 같은 placeholder를 넣어 공유할 수 있다.

## Docker 관련

- Docker를 사용하는 이유는 무엇인가?

  → Docker를 사용하면 개발자의 로컬 환경에 따라 실행 결과가 달라지는 문제를 줄일 수 있다. Python 버전, 패키지 설치, 실행 명령어 등을 Dockerfile에 정의해두면 같은 환경에서 백엔드를 실행할 수 있다. 또한 Render 같은 배포 환경에서도 Docker 이미지를 기반으로 서비스를 안정적으로 배포할 수 있다.

- Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

  → `EXPOSE`는 컨테이너 내부에서 사용할 포트를 문서적으로 선언하는 역할을 한다. 이 프로젝트에서는 FastAPI가 8000번 포트를 사용하므로 `EXPOSE 8000`을 작성했다. `CMD`는 컨테이너가 실행될 때 자동으로 실행할 명령어를 지정한다. 이 프로젝트에서는 `uvicorn main:app --host 0.0.0.0 --port 8000` 명령으로 FastAPI 서버를 실행하도록 설정했다.
